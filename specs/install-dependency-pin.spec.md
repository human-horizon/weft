# Установка Weft без служебного workspace

## Контекст

`weft install` создаёт `pnpm-workspace.yaml` только для обхода отсутствующей версии транзитивной AWS-зависимости. Пользовательский проект не должен получать workspace-конфигурацию ради внутренней проблемы зависимости Weft.

## Цель

Обеспечить успешный `pnpm install` внутри `.lore/weft/` без создания `.gitignore` и `pnpm-workspace.yaml`.

## Что изменится

1. `package.json` — зависимость от Pi будет зафиксирована на проверенной рабочей версии.
2. `src/cli.ts` — команда `weft install` перестанет создавать `pnpm-workspace.yaml`.
3. `specs/cli.spec.md` и `README.md` — структура установки будет приведена к фактическому поведению.
4. Тесты — будет зафиксирован состав создаваемых файлов.

## Детали реализации

1. Проверить чистую установку выбранной версии `@earendil-works/pi-coding-agent` без lock-файла.
2. Использовать только подтверждённую комбинацию зависимостей.
3. Оставить создание `.lore/weft/package.json` и `.lore/weft/pipelines/`.
4. Запускать `pnpm install` с рабочей директорией `.lore/weft/`.
5. Не создавать `.gitignore` и `pnpm-workspace.yaml`.

## Критерии приёмки

- [x] Чистая установка упакованного пакета Weft не запрашивает отсутствующую AWS-зависимость.
- [x] `weft install` не создаёт `.gitignore`.
- [x] `weft install` не создаёт `pnpm-workspace.yaml`.
- [x] `pnpm install` выполняется внутри `.lore/weft/` и завершается успешно.
- [x] Сборка, статическая проверка и тесты проходят.
